## 1. Ship name bbox — privacy & necessity

**Context:** Currently the user can draw a bbox labelled `Ship Name` over the ship name text
in a screenshot. The bbox position is saved in `annotations.json` and the text inside is read
by OCR. Ship names are personal data (player character names).

**Open questions:**
- Do we need the bbox at all? The ship name is already extracted by OCR from the top-band scan
  (`TextExtractor.extract_ship_info`) without a bbox. If OCR works reliably without it, the
  `Ship Name` slot type can be removed entirely from WARP CORE.
- If we keep it: the bbox should be saved for layout purposes (helps confirm the Y-position of
  the name row) but the text content must never be uploaded to HF or the backend.
- **Action required:** Audit `SyncWorker` — verify that crop PNGs for `Ship Name` slots are
  excluded from HF upload (or that the slot is already in `NON_ICON_SLOTS` and therefore no
  crop is saved/uploaded).

**Decision needed:** Keep bbox (position only, no content) / remove slot type entirely.

## 2. User-drawn slot label bboxes (Fore Weapons, Deflector, …)

**Context:** WARP CORE lets users draw bboxes for icon slots. Currently the intent is to mark
where icons are. A related idea: let users draw bboxes around the **slot label text** (e.g.
"Fore Weapons", "Deflector") to provide ground truth for layout detection.

**Open questions:**
- Is this worth the UI complexity? Slot label positions can already be inferred from icon
  positions once icons are confirmed.
- If implemented: label bboxes would be a new slot type (e.g. `Label: Fore Weapons`) — they
  feed only into `anchors.json` / community anchors (P11), not into icon training.
- Alternative: skip label bboxes entirely; P11 community anchors cover this use case without
  requiring users to annotate text.

**Decision needed:** Implement label bbox annotation / skip in favour of P11.

## 3. One-per-screen enforcement for Ship Name / Ship Tier / Ship Type

**Context:** `Ship Name`, `Ship Tier`, `Ship Type` are `NON_ICON_SLOTS` — only one per
screenshot makes sense. It is unclear whether WARP CORE currently prevents the user from
drawing a second bbox of the same type on the same screenshot.

**Verify:**
- Check `TrainingDataManager.add_annotation()` — does it reject or overwrite a duplicate
  `NON_ICON_SLOT` bbox on the same screenshot?
- Check the duplicate-overlap warning in `trainer_window.py` (`_on_accept`) — does it
  trigger for same-type NON_ICON_SLOTS?

**If not enforced:** Add a guard in `_on_accept` / `_on_bbox_drawn`:
```python
if slot in NON_ICON_SLOTS:
    existing = [a for a in confirmed_anns if a.slot == slot]
    if existing:
        # overwrite or warn — do not create second bbox
```

**Possible extension:** Allow more than one Ship Name bbox if a screenshot shows multiple
ships (e.g. fleet bank screen). Unlikely in practice — defer until a real use case arises.
